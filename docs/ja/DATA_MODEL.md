### 1.2 Structured CSV 代替正規化フォームとして

セマンティック・インターチェンジのStructured CSVは代替手段として提案される**Coddの関連正規化**. コッドスタイルの正規化は、データを複数の関係に分解し、各事実を一度保存し、プライマリおよび外部キーを介して関係を接続することにより、更新異常を削除します。 これは、リレーショナルデータベース内で有効であるが、インターチェンジパッケージは、いくつかのテーブル、そのキー、およびその参加ルールを運び、調整する必要があります。

Structured CSV は、冗長コピーなしで、各事実の1つの認証的発生を同一の重要な目的を維持しますが、1つの物理的なテーブルで正規関係を表します。 疎な行 の所有権と ディメンション列 を通じて、論理的な分離を維持しながら、そうでなければいくつかの正規化テーブルになるものを組み合わせたものです。

|地域正規化 | 階層的 Structured CSV |
| --- | --- |
|各団体・リピートグループ・リピートグループ関連1組行スコープ 各リピートクラス|
|プライマリキーは1つのリレーションで行を識別します | ディメンション 値がクラスの発生を識別します |
|外部キーは、その親に関連する子供を関連付ける | アニストー ディメンション列 は、その親行に子行を関連付ける |
|複数の物理テーブルを組み合わせる | マッチング ディメンション 1つのCSV内の階層を再構築するパス |
|親子関係に親の値がコピーされていない | 親属性は親行に残り、子行にコピーされていない |

したがって、**1 つの CSV ファイルが 1 つの非正規化の関係を意味しない**. それは複数の論理的に正常化された行スコープsのための容器です。 ディメンション リレーショナルテーブルを分離する関係を外部キーに表現する。 消費者は、各ディメンションが所有する行を選択して、個々の論理テーブルを再構築したり、各祖先の行をグループ化することにより、完全な階層を再構築することができますディメンションパス。

このモデルは、リレーショナルな通常のフォームやデータベースの制約を無効にしません。 耐久性、シンタックスニュートラルデータ交換のための異なる正規化とシリアライズ機構を提供します。複数の論理テーブルは、事実を複製することなく1つのファイルに結合され、その関係は明示的です。

## 4. Structured CSV 階層

### 4.1 ディメンション列s

Structured CSV は ディメンション列 でそれぞれ 反復クラス を表します。 ディメンション 値は、現在の祖先 適用範囲 内の発生を識別します。 属性カラムは値を保持します。空白のセルは、行がその属性を所有していないことを意味します。

### 4.2 単数子クラス

お問い合わせ**dBbb**繰り返されるのは単数です**dAaa**両方のクラスの値が同じ行を共有します。

```csv
dAaa,dBbb,a1,a2,b1,b2,b3
1,,a1V1,a2V1,b1V1,b2V1,b3V1
```

単数の子は、最も近い繰り返しの祖先の列に所属し、独立した発生行を受けません。

### 4.3 反復子クラス

お問い合わせ**dBbb**繰り返されると、親属性は親行を占有し、すべての子の発生は独立した行を占めます。

```csv
dAaa,dBbb,a1,a2,b1,b2,b3
1,,a1V1,a2V1,,,
1,1,,,b1V1,b2V1,b3V1
1,2,,,b1V2,b2V2,b3V2
```

最初の繰り返しの子を親行にマージするため、次のコンパクトなフォームは無効です。

```csv
dAaa,dBbb,a1,a2,b1,b2,b3
1,1,a1V1,a2V1,b1V1,b2V1,b3V1
1,2,,,b1V2,b2V2,b3V2
```

### 4.4 所有権の派生

コンバーターは、バインディング行から派生した列の所有権を生成します。**タイプ=C**クラス**multiplicity**と各属性**semantic_path**. 単数の子が最も近い繰り返された祖先の 行スコープ を継承する。 繰り返された子供は新しいディメンションを所有し、発生1から独立した行を出す。 このルールは、すべての深さで再帰的に適用されます。

### 4.5 疎な行 契約

行は意図的にスパースされます。 親行には、適用範囲 が所有する値のみが含まれています。繰り返し子行には、その祖先とその子が所有する値(適用範囲)を識別するために必要な ディメンション が含まれています。 Phase 2 をマージすると、現在の祖先値をターゲットバインディングが必要とする場所にのみ、対象レコードに格納します。

## 7. 監査の継続性とデータ境界

ADS と ADC は、通常 ERP 環境で動作します。 ユーザーの活動、説明の詳細、終了エントリ、および年間レポートの準備は、個々の インボイス 外部に記録されます。 その結果、 インボイス 単独では、各監査、投稿、承認、またはそれらのターゲットモデルのフィールドを報告することはできません。

UADC は、その境界を明示します。 インボイス がサポートするすべての情報をマッピングし、未知のターゲットフィールドを既知のギャップとして保存し、独立したインターフェイスファイル間で安定したセマンティック識別子と階層を保持します。 したがって、組織は、LHMとバインディング機構を耐久性のあるセマンティックレイヤーとして使用しながら、独自の入力と出力フォーマットを維持することができます。 ERP 操作履歴と管理されたバインディングバージョンと共に、長期的、再現性、監査済みデータ保持の基礎を提供します。

## 15. 統合スクリプト処理ガイド

### UADC スクリプト処理ガイド

#### 処理モデル

UADC PoC は 2 つの段階に処理を分けます。

```
Source syntax XML
  -> syntax binding
  -> UADC Structured CSV

UADC Structured CSV
  -> target syntax binding or semantic binding
  -> XBRL GL, PSV, or CSV target view
```

Phase 1 は共通階層 Structured CSV を作成します。 Phase 2 は Structured CSV を消費し、対象ビュー を生成します。

コアの考え方は、バインディング行が3つのことを接続することです。

- 著名な場所**semantic_path**;
- 構文の場所は、**xpathの**XML または XBRL XML が関与している場合
- a Structured CSV 列, として書かれて**構造 csv column**またはから得られる**semantic_path**.

クラス行を使う**タイプ=C**. 事実上の行は使用します**タイプ=A**. クラス行はツリーを定義し、 行スコープ を繰り返します。 Fact 行は値を定義します。

#### 意味パス および XPath

**semantic_path**共通UADCのセマンティックモデルの用語の位置を識別します。

```

```

最後のセグメントがセマンティックな親クラスを識別する前のパス。 通常、最終セグメントは、Structured CSV 値の列になります。

```

$.invoice.invoiceLine.invoiceLineIdentifier -> InvoiceLineIdentifier
```

**xpathの**具体的なXMLまたはXBRL XML要素または属性を識別します。

```

いつか**xpathの**属性で終了します。**エレメント**所有者の要素のまま。

```

対象ビュー.
2. **row 適用範囲 group(...)**インボイス レベルの変換または
最も深い非インデックス化された繰り返しターゲット 適用範囲.
3. **row repeat indices(...)**そして、**リピート group current(...)**メンテナンス
明示的にインデックスされた意味バインディングsで使用されるゼロベースのインデックス。
4. **変換 rows(...)**インボイスレベルのマージまたは
**変換 repeated group rows(...)**.
5. **write target file(...)**選択したバインディング・オーダー・カラムを書き込みます
デリミター。
6. **output format options(...)**PSV または CSV のデフォルトとコマンドラインを解決
オーバーライド。
7. **output path(...)**そして、**バインディング target stem(...)**ターゲットファイルを導き出す
所在地と名前。
8. **変換 ファイル(...)**1つの入力ファイル変換を実行します。
**メイン(...)**ディレクトリの入力を拡張します。**入力ファイル(...)**.
