# 決定: xBRL-CSV メタデータとスキーマ - 応用 ラウンドトリップ XML

## コンテンツ

コンバータは、もともとUADC固有のJSONマッピングファイルを生成しました。 初期開発では有用だったが、ArelleでxBRL-CSVと直接読み込むことができない。 PoC は、CSV を UBL Invoice に再構築できる、スキーマ・バリド・シンタックスを持つ XML に構造化した証明書も必要です。

## デコレーション

1. 構文バインディング変換はxBRL-CSVメタデータJSONではなく書きます
UADC 固有のメタデータ形式。
2. メタデータは**documentInfo**,**tables**,**tableTemplates**を使用します。
3. 生成された**en16931-oim**タクソノミ スキーマを xBRL-CSV と参照します。
OIM エントリーポイント。
4. Arelle**loadFromOIM**検証は、生成されたメタデータの回帰チェックです。
5. リバースコンバージョンの再構築 UBL Invoice 構造化されたCSVからXML
LHM XPath および 構文バインディング ルール。
6. リバースコンバージョンは、EN 16931 BT ではない UBL 必須サポート値を追加します。
スキーマの有効性(**cac:TaxScheme/cbc:ID**)および欠落(**cbc:ChargeIndicator=false**)などの値、価格の許容コンテキスト。
7. 逆変換はサポートされるUBLの子供の要素の順序を前に正規化します
XML を書く。
8. UBL 2.1 スキーマ検証は、再生された XML の回帰チェックです。

## コンシーケンス

- Structured CSV 出力は xBRL-CSV タクソノミ に直接結ばれ、
Arelle で読み込まれる。
- PoC は、シンタックス 必須 サポート値からセマンティックな値を区別します。
- ラウンドトリップXMLは、XMLと同じバイト単位であることが期待されていません。
しかし、境界値を保存し、UBLスキーマ制約を満たす必要があります。
- ローカル UBL 2.1 XSD のキャッシュは生成されたか、ローカルの検証データであり、
コミットしないでください。
